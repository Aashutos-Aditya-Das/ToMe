## 1. EVIDENCE ENGINE MISSION

The mission of the Evidence Engine is to answer one critical question: **"Why does ToMe believe this claim is true?"**

ToMe is not a black-box AI chat interface; it is an intelligence compiler. Every piece of architectural intelligence (RIS) generated by ToMe must be mathematically tethered to the physical reality of the codebase (Repository Model) or a historical human override. The Evidence Engine is the connective tissue that makes this traceability possible.

## 2. EVIDENCE PHILOSOPHY

1. **Hallucination is Guilt Without Evidence:** An LLM assertion that lacks a structural pointer is immediately classified as a hallucination and purged.
2. **Probability Requires Proof:** We do not accept flat JSON attributes (e.g., `isAuthentication: true`). We require a `Claim` object containing the source, the method of derivation, and the confidence level.
3. **Immutability of Human Truth:** Human assertions override all probabilistic inferences. They hold infinite confidence and serve as anchor nodes for future LLM deduction.

## 3. WHY TRACEABILITY MATTERS

Without traceability, AI-generated intelligence decays the moment the code changes. If the RIS claims "The Payment Service uses Stripe," and an engineer refactors the codebase to use Braintree, ToMe must know exactly *which* file triggered the original Stripe claim. When that file is deleted, the Evidence Engine alerts the RIS that the claim's foundation has crumbled, triggering a probabilistic recalculation.

---

## 4. EVIDENCE ARCHITECTURE

The Evidence Engine sits as an intermediary layer operating during Phase 2 (LLM Hydration) and Phase 4 (Reconciliation) of the RIS lifecycle.

```text
+---------------------+      +---------------------+      +---------------------+
|                     |      |                     |      |                     |
|  RIS GRAPH (Intent) |<---->|   EVIDENCE GRAPH    |<---->|  REPO MODEL (Code)  |
|                     |      |                     |      |                     |
+---------------------+      +---------------------+      +---------------------+
      Domain: Billing             Claim: Uses Stripe          Class: StripeClient
      Capability: Charge          Confidence: 0.92            Function: charge()
```

## 5. EVIDENCE LIFECYCLE

1. **Extraction:** The Prompt System extracts a semantic claim and binds it to a Repository Model Node ID.
2. **Validation:** The Evidence Engine verifies that the Repository Model Node actually exists in the current parsing state.
3. **Scoring:** The Engine calculates a `ConfidenceScore` based on the density and quality of the evidence.
4. **Binding:** The `Claim` is permanently attached to the `RIS Node`.
5. **Decay/Invalidation:** When `tome update` detects code changes, Evidence Edges pointing to modified/deleted Code Nodes are severed, marking the parent `Claim` as `DIRTY`.

---

## 6. EVIDENCE GRAPH MODEL

The Evidence Engine operates its own sub-graph. 
- **Nodes:** Represent pieces of evidence (e.g., a specific file, a specific human assertion).
- **Edges:** Represent the deductive leaps (e.g., "Node A *implies* Claim B").

## 9. CLAIM MODEL

The `Claim` is the fundamental wrapper around every semantic attribute in the RIS.

```typescript
export interface Claim<T> {
  id: string;
  risNodeId: string; // The Domain, Service, or Capability this claim belongs to
  attributeName: string; // e.g., 'usesDatabase' or 'businessRule'
  value: T; // The actual semantic data (string, boolean, array)
  status: ClaimStatus;
  derivation: DerivationMethod;
  confidence: ConfidenceScore;
  provenance: ProvenanceRecord;
  evidenceEdges: EvidenceEdge[]; // The graph connections to proof
}
```

---

## 10. CLAIM PROVENANCE ARCHITECTURE

Provenance guarantees auditability. A `ProvenanceRecord` tracks the exact conditions under which a `Claim` was born. If an LLM prompt is later found to be defective, ToMe can query the Evidence Graph for all claims generated by `promptVersion: "v1.2"`, mark them as `DIRTY`, and re-evaluate them.

---

## 11. EVIDENCE COLLECTION PIPELINE

During Phase 2 (LLM Hydration), the Prompt System is strictly instructed to return data matching the `Claim` schema.

**Example LLM Output Segment:**
```json
{
  "claims": [
    {
      "attributeName": "TechStack",
      "value": "PostgreSQL",
      "derivation": "LLM_INFERRED",
      "ConfidenceLevel": 0.9,
      "supportingNodes": ["RepoNode:File:src/db/connection.ts", "RepoNode:Dependency:pg"]
    }
  ]
}
```
The Engine parses this, verifies that `src/db/connection.ts` and `pg` exist in the Repository Model, creates `EvidenceNode` and `EvidenceEdge` objects, and attaches them to the `Claim`.

---

## 12. EVIDENCE AGGREGATION LOGIC

Multiple `EvidenceNodes` strengthen a `Claim`.
If `Claim(A)` is supported by exactly one `EvidenceNode(Class: StripeClient)`, it is moderately confident.
If `Claim(A)` is supported by `EvidenceNode(Class: StripeClient)`, `EvidenceNode(Dependency: @stripe/stripe-js)`, and `EvidenceNode(EnvVar: STRIPE_SECRET)`, the evidence is *aggregated*.

---

## 13. EVIDENCE SCORING SYSTEM

Evidence nodes have intrinsic weights:
* **Dependency Node:** Weight 0.8 (Very strong proof of intent).
* **Environment Variable Node:** Weight 0.7 (Strong proof of integration).
* **Class/Method Node:** Weight 0.6 (Moderate proof, subject to naming anomalies).
* **File Name Node:** Weight 0.4 (Weak proof).
* **Human Assertion:** Weight 1.0 (Absolute proof).

---

## 14. CONFIDENCE CALCULATION FRAMEWORK

```text
ConfidenceScore min(1.0, ConfidenceLevel * (1 + Sum(EvidenceNode.weight * Edge.weight)))
```

**Formula Rules:**
1. If `Derivation == HUMAN_ASSERTED`, `ConfidenceScore 1.0`.
2. If `Derivation == OBSERVED_STRUCTURAL`, `ConfidenceScore 1.0`.
3. If `Derivation == LLM_INFERRED`, the score caps at `0.98` (to reflect inherent AI probabilistic doubt) unless human-validated.
4. If a Claim has zero verified Evidence Edges, its confidence drops to `0.0` and it is immediately purged.

---

## 15. STRUCTURAL EVIDENCE VS SEMANTIC EVIDENCE

* **Structural Evidence:** Hard facts from the Repository Model. (e.g., "Function `login` invokes Function `hashPassword`").
* **Semantic Evidence:** Output from previous LLM reasoning stages. (e.g., "The `login` function is part of the `AuthDomain`"). 

The Evidence Engine primarily links Semantic Claims to Structural Evidence to ground the AI in reality.

---

## 16. OBSERVED FACTS VS INFERRED FACTS

* **OBSERVED:** A fact mathematically proven by the Tree-sitter parser. (e.g., "This project uses React").
* **INFERRED:** A fact deduced by the LLM. (e.g., "This project uses a Micro-Frontend architecture").

The engine treats Observed facts as immutable graph anchors. Inferred facts are fragile and subject to recalculation.

---

## 17. HUMAN ASSERTED FACTS

When a developer edits `memory.md` directly, ToMe's update process parses the Markdown diff. It identifies the modified text, locates the underlying `Claim` in the internal JSON state, and transitions its `DerivationMethod` to `HUMAN_ASSERTED`.

This is the most critical feature of the Evidence Engine. It guarantees that ToMe is a tool that serves the developer, not an AI that fights them.

---

## 18. EVIDENCE VALIDATION RULES

1. **The Phantom Node Rule:** If the LLM generates a Claim supported by `RepoNode:Class:PaymentService`, but `PaymentService` is not present in the Repository Skeleton, the Evidence Edge is rejected.
2. **The Minimum Support Rule:** A Claim must have at least one valid Evidence Edge, or it is discarded as hallucinated noise.

---

## 19. EVIDENCE INTEGRITY RULES

When the Evidence Graph is updated, the engine ensures no loops exist (e.g., Claim A is proven by Claim B, which is proven by Claim A). Evidence must ultimately bottom out into Structural Nodes or Human Assertions.

---

## 20. EVIDENCE MUTATION RULES

1. **LLMs cannot mutate HUMAN_ASSERTED claims.**
2. If the LLM believes a `HUMAN_ASSERTED` claim is no longer true (due to sweeping code changes), it generates a `SuggestedMutation`. The CLI warns the user: *"Your memory explicitly states you use Redis, but all Redis dependencies have been removed. Should I update this?"*

---

## 21. EVIDENCE VERSIONING

Evidence Edges maintain a checksum of the Structural Node they point to. If `src/auth.ts` changes, its checksum changes. During `tome update`, the engine identifies all Evidence Edges pointing to stale checksums and flags the connected Claims as `DIRTY`.

---

## 22. EVIDENCE PERSISTENCE STRATEGY

The Evidence Graph is heavy. We do not expose it to the user in `.tome/architect.md` because it would pollute the document with JSON pointers.

Instead, the Evidence Graph is persisted in a hidden state file: `.tome/.ris-state.json` (or locally in the system cache). This allows ToMe to maintain full graph continuity between CLI invocations without exposing the math to the human.

---

## 23. RIS ↔ EVIDENCE INTEGRATION

The RIS is no longer a collection of strings. 
```typescript
// OLD RIS:
export interface Service {
  name: string;
  techStack: string[];
}

// NEW INTEGRATED RIS:
export interface Service {
  name: Claim<string>;
  techStack: Claim<string[]>;
}
```
Every property on every RIS entity is a `Claim`.

---

## 24. REPOSITORY MODEL ↔ EVIDENCE INTEGRATION

The Repository Model remains ignorant of Evidence. It simply provides the globally unique IDs (e.g., `RepoNode:Function:calculateTax`) that the Evidence Edges point to. It serves as the immutable bedrock of the graph.

---

## 25. PROMPT SYSTEM ↔ EVIDENCE INTEGRATION

The Prompts are architected to extract evidence.
*Prompt Instruction:* "For every architectural pattern you identify, you MUST return a `supportingNodes` array containing exact Node IDs from the provided Repository Skeleton."

---

## 26. ARTIFACT GENERATION ↔ EVIDENCE INTEGRATION

When `MemorySerializer` generates `architect.md`, it unwraps the `Claim` objects. 
It formats the Markdown cleanly. 
*Future Extension:* If a Claim has a ConfidenceScore < 0.6, the serializer might append an alert: `> [!WARNING] AI Inference Confidence Low`.

---

## 27. EVIDENCE COMPRESSION STRATEGY

Over years of development, thousands of orphaned claims and obsolete evidence nodes will bloat `.ris-state.json`.
* **Garbage Collection:** Any `Claim` marked `ORPHANED` for more than 3 `tome update` cycles is physically deleted from the graph.

## 30. FINAL ENGINEERING VERDICT

**Conclusion:**
The Evidence Engine solves the core problem of AI developer tools: **Trust**.

By ensuring every semantic inference is tightly bound to a physical reality (the code), and by providing a mechanism for human overrides (`HUMAN_ASSERTED`) that completely freezes LLM hallucinations, we guarantee that ToMe's intelligence will never silently drift into fantasy. 

The integration of `Claim`, `ConfidenceScore`, and `ProvenanceRecord` into the RIS represents a rigorous, enterprise-grade architecture ready for implementation.
